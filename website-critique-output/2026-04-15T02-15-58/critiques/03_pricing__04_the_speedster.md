## The Performance Audit

Here is your prioritized battle plan. Fixes are ranked by their impact on perceived speed and Core Web Vitals.

**1. LCP Element Optimization: The Hero Heading**
*   **What's wrong:** Your Largest Contentful Paint (LCP) element is the text block: *"Stop losing $74K a year in silent member attrition..."* Because it uses a custom Serif font, the browser cannot paint this text until the font file is downloaded, creating a Flash of Invisible Text (FOIT).
*   **Impact:** Delays your primary LCP metric by 300–800ms depending on network speed.
*   **The Fix:** Add a `<link rel="preload" href="/fonts/your-serif-font.woff2" as="font" type="font/woff2" crossorigin>` tag to your `<head>` for the specific weight used in that H1. Add `font-display: swap` to your CSS `@font-face` declaration to ensure system fonts render immediately while the custom font loads.

**2. INP Risk: ROI Calculator Slider State**
*   **What's wrong:** The ROI calculator features three interactive sliders ("TOTAL MEMBERS", "AVG ANNUAL DUES", "ANNUAL TURNOVER RATE"). In React/Vue, binding a slider's `onChange` event directly to a complex chart re-render often blocks the main thread, causing slider drag lag (poor INP).
*   **Impact:** A sluggish, "sticky" feeling when dragging the sliders, easily pushing INP over the 200ms failing threshold.
*   **The Fix:** Decouple the slider handle movement from the chart recalculation. Use React's `useTransition` to mark the chart update as a non-urgent state, ensuring the slider UI stays locked to the user's cursor at 60fps, while the chart updates milliseconds later.

**3. JS Payload Bloat: Heavy Charting Libraries**
*   **What's wrong:** The "DUES PROTECTED" line chart. It is tempting to install Recharts, Chart.js, or D3.js to render this single, simple line graph. 
*   **Impact:** Adding a heavy visualization library for one chart can add 80KB–150KB of blocking JavaScript to your main bundle, delaying Time to Interactive (TTI).
*   **The Fix:** Hand-roll this chart using pure inline SVG generated by your front-end framework, or use a micro-library like `uPlot` (under 5KB). Do not ship a massive visualization engine for a static-looking line and a dot.

**4. CLS Risk: Serif to Sans-Serif Font Reflow**
*   **What's wrong:** When implementing `font-display: swap` (as recommended in step 1), the browser will display a fallback font (like Times New Roman) before snapping to your custom Serif. If the x-heights or kerning differ significantly, the text will shift, pushing down the "5 of 7 founding-partner clubs..." paragraph.
*   **Impact:** Unnecessary Cumulative Layout Shift (CLS) penalty in the hero section.
*   **The Fix:** Use CSS font metrics overrides (`size-adjust`, `ascent-override`, `descent-override`) in your `@font-face` declaration to mathematically scale the fallback system font so it takes up the exact same physical pixel dimensions as your custom web font.

**5. Asset Optimization: Inline SVG Checkmarks**
*   **What's wrong:** The pricing cards ("Signals", "Signals + Actions") contain over 15 orange checkmark icons. If these are loaded as separate `<img>` tags or external files, they clog the browser's download queue.
*   **Impact:** Unnecessary HTTP requests that contend with more critical assets (like JS and fonts).
*   **The Fix:** Inline the SVG path directly into the HTML payload as a reusable React/Vue component, or use an SVG `<use>` sprite map. This guarantees the icons render instantly with the HTML DOM.

**6. JavaScript Execution: Native HTML FAQ Accordions**
*   **What's wrong:** The "Things GMs ask us" section uses accordions. Developers frequently use heavy JavaScript state to handle the open/close toggle of these elements.
*   **Impact:** Wasted main thread execution time during page load to attach event listeners to elements far below the fold.
*   **The Fix:** Build the FAQ purely with native HTML `<details>` and `<summary>` tags. This pushes the toggle logic to the browser engine, requiring zero custom JavaScript and achieving zero INP latency.

**7. Code Splitting: Deferring the Pricing Cards**
*   **What's wrong:** The massive pricing section is far below the initial viewport. Loading the logic, SVGs, and specific CSS for this section during the initial page load delays the hero and calculator functionality.
*   **Impact:** Slower initial page load and larger initial JS bundle.
*   **The Fix:** Implement component-level code splitting. Lazy-load the `<PricingSection />` and `<FAQSection />` components so their JavaScript chunks are only fetched after the browser goes idle or as the user begins scrolling.

**8. Infrastructure: Edge Caching for Static Assets**
*   **What's wrong:** (Contextual based on Vercel hosting). Failing to explicitly declare long-lived caching headers for your compiled assets.
*   **Impact:** Returning users must re-download JavaScript and CSS, slowing down subsequent page views.
*   **The Fix:** Ensure your Vercel configuration (`vercel.json` or Next.js config) sets `Cache-Control: public, max-age=31536000, immutable` for all static assets (fonts, hashed JS/CSS files).

**9. Perceived Performance: The Calculator Skeleton**
*   **What's wrong:** If you code-split the ROI calculator to save initial bundle size, a blank white space will appear where the calculator should be until the JS downloads and executes.
*   **Impact:** A jarring layout pop-in and a feeling that the site is broken or slow.
*   **The Fix:** Server-Side Render (SSR) the initial state of the calculator (e.g., slider inputs at default positions, empty graph axis) using HTML/CSS only. This serves as a perfect structural skeleton while the interactive JavaScript "hydrates" silently in the background.

**10. Third-Party Scripts: Marketing Pixels**
*   **What's wrong:** As a marketing page, you will eventually add Google Analytics, HubSpot, or LinkedIn tracking pixels. Putting these synchronously in the `<head>` is performance suicide.
*   **Impact:** Blocks the main thread, destroys LCP, and ruins INP.
*   **The Fix:** Load all third-party tracking scripts with the `defer` attribute, or better yet, use a Web Worker solution like Partytown to execute them off the main thread entirely.
